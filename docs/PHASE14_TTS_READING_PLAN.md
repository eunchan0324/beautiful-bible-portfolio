# Phase 14: TTS 성경 듣기 기획

> 이 문서는 #187 TTS 성경 듣기 기능의 조사 결과와 MVP 방향을 정리한다.
> 실제 구현은 브라우저 내장 TTS 기반 프론트엔드 MVP부터 시작하고, 서버 TTS나 오디오 성경 연동은 후속 Phase로 분리한다.

## 배경

현재 Beautiful Bible의 성경 읽기 화면은 백엔드 API가 아니라 프론트엔드의 정적 성경 데이터(`/bible.json`)를 읽는다.
따라서 Render 무료 서버가 잠들어 있어도 기본 성경 읽기는 바로 가능하다.

TTS도 이 장점을 유지하는 방향으로 시작한다.
처음부터 서버에서 음성을 생성하거나 오디오 파일을 저장하면 비용, 라이선스, 저장소, 캐싱, 콜드스타트 문제가 함께 생긴다.
Phase 14의 목표는 완성형 오디오 성경을 만드는 것이 아니라, 사용자가 성경 본문을 눈으로 읽기 어려운 순간에도 현재 장을 들을 수 있는 최소 기능을 검증하는 것이다.

## 타 성경 앱 조사 요약

대형 성경 앱은 대부분 즉석 TTS보다 녹음된 오디오 성경 UX에 가깝다.

- YouVersion Bible App은 성경을 읽고 듣는 기능을 제공한다.
- BibleGateway Audio는 책, 장, 오디오 판본을 선택하고 재생 속도, 타이머, 연속 재생을 제공한다.
- Bible Brain은 텍스트, 오디오, 비디오 성경 데이터를 API로 제공한다.

공통 UX는 다음과 같다.

- 현재 장 단위 재생
- 재생, 일시정지, 다음 장, 이전 장
- 재생 속도 조절
- 일부 앱의 타이머, 연속 재생, 오프라인 저장
- 오디오와 본문 위치의 느슨한 동기화

Beautiful Bible MVP에서는 이 전체를 따라가지 않는다.
오디오 성경은 UX 품질은 좋지만 라이선스와 데이터 운영 비용이 크다.
따라서 먼저 브라우저 내장 TTS로 작은 듣기 경험을 검증한다.

참고 출처:

- https://www.bible.com/app
- https://www.biblegateway.com/audio/
- https://www.faithcomesbyhearing.com/audio-bible-resources/bible-brain
- https://developer.mozilla.org/en-US/docs/Web/API/SpeechSynthesis
- https://cloud.google.com/text-to-speech/docs
- https://docs.aws.amazon.com/polly/latest/dg/what-is.html

## 선택지 비교

### 1. 브라우저 내장 TTS

Web Speech API의 `SpeechSynthesis`를 사용해 브라우저가 현재 본문을 읽게 한다.

장점:

- 서버 호출이 없다.
- Render 콜드스타트와 무관하다.
- 별도 비용이 없다.
- 현재 정적 성경 데이터 구조를 그대로 활용할 수 있다.
- MVP 구현 범위가 작다.

단점:

- 기기와 브라우저마다 음성 품질이 다르다.
- iOS, Android, Chrome, Safari의 동작 차이가 있을 수 있다.
- 백그라운드 재생이나 잠금화면 제어는 안정적으로 보장하기 어렵다.
- 성경 고유명사 발음이 자연스럽지 않을 수 있다.

MVP 추천안이다.

### 2. 클라우드 TTS 음성 생성

Google Cloud Text-to-Speech, Amazon Polly, OpenAI TTS 같은 서비스를 사용해 서버에서 음성을 생성한다.

장점:

- 음성 품질과 일관성이 좋다.
- 캐싱하면 모든 기기에서 같은 음성을 제공할 수 있다.
- 향후 오디오 플레이어 UX를 더 안정적으로 만들 수 있다.

단점:

- 비용이 발생한다.
- 백엔드 API, 저장소, 캐시 키, 음성 파일 관리가 필요하다.
- 현재 무료 서버 운영 조건과 맞지 않는다.
- 성경 본문을 음성으로 생성하는 사용권 검토가 필요하다.

후원 또는 유료 서버 전환 이후 검토한다.

### 3. 오디오 성경 파일 또는 API 연동

녹음된 오디오 성경을 직접 보유하거나 외부 API로 연동한다.

장점:

- 사용자 입장에서 가장 자연스러운 듣기 경험을 제공할 수 있다.
- 긴 시간 듣기, 연속 재생, 백그라운드 재생에 적합하다.

단점:

- 라이선스 확인이 가장 중요하다.
- 한국어 오디오 성경은 무료 사용 가능한 자료를 찾기 어렵다.
- 파일 용량, 스트리밍, 저장소, 네트워크 비용이 커질 수 있다.

MVP에서는 제외한다.

## MVP 방향

Phase 14 MVP는 프론트엔드에서만 동작하는 `현재 장 듣기` 기능으로 제한한다.

포함한다.

- 성경 읽기 화면에서 현재 장 듣기
- 한국어 본문 우선 지원
- 재생, 일시정지, 정지
- 재생 속도 선택
- 절 단위로 끊어서 읽기
- 읽는 중인 절 표시
- 페이지 이동 또는 장 변경 시 자동 정지
- 브라우저가 TTS를 지원하지 않을 때 안내 문구 표시

포함하지 않는다.

- 서버 TTS 생성
- 음성 파일 저장
- 백그라운드 재생
- 잠금화면 미디어 컨트롤
- 오프라인 오디오 저장
- 오디오 성경 API 연동
- 여러 음성 선택 UI
- 통독 전용 자동 연속 재생
- WEB 영어 본문 동시 낭독

## UX 방향

TTS는 "현재 장을 듣는 행동"이다.
처음에는 장 제목 영역의 `AI 장 설명` 버튼 옆에 듣기 버튼을 두는 방안도 검토했지만, Beautiful Bible의 성경 읽기 화면은 스크롤 방식이므로 하단 컨트롤바에 배치하는 편이 더 적합하다.

사용자는 장 중간이나 맨 아래까지 내려간 상태에서도 듣기를 시작하거나 멈추고 싶을 수 있다.
상단에만 버튼이 있으면 다시 위로 올라가야 하므로 접근성이 떨어진다.

따라서 MVP에서는 기존 하단 컨트롤바에 `듣기` 버튼을 추가한다.

```text
현재:
화면 모드 / 역본 / 글씨 크기

변경:
화면 모드 / 듣기 / 역본 / 글씨 크기
```

`듣기`는 본문을 소비하는 방식이고, `역본`은 본문 표시 방식이다.
그래서 `화면 모드` 다음에 `듣기`를 배치하고, 이후에 `역본`, `글씨 크기`를 두는 방향을 우선 검토한다.

재생 상태는 하단 버튼 자체에서 먼저 표현한다.

```text
대기 상태: 듣기
재생 상태: 일시정지
일시정지 상태: 이어듣기
```

MVP에서는 별도의 큰 오디오 플레이어를 만들지 않는다.
필요하다면 하단 컨트롤바 위에 작은 상태 표시만 추가한다.

```text
창세기 1장 듣는 중     1x
```

이 상태 표시는 본문을 가리지 않도록 작게 유지하고, 재생 중일 때만 노출한다.

통독 화면에서도 같은 성경 읽기 화면을 사용하므로, TTS는 통독 모드에서도 동작할 수 있다.
다만 MVP에서는 "현재 장 듣기"까지만 제공하고, 통독의 여러 장을 자동으로 이어 읽는 기능은 제외한다.

## 기술 설계 초안

브라우저 API는 다음 흐름으로 사용한다.

```text
현재 장 verses
-> 절 단위 텍스트 배열 생성
-> SpeechSynthesisUtterance 생성
-> lang = ko-KR
-> rate = 사용자 선택 속도
-> onend에서 다음 절 재생
-> 현재 읽는 verse_key 상태 업데이트
```

구현 단위는 다음과 같이 나눌 수 있다.

```text
useSpeechSynthesisReader
- TTS 지원 여부 확인
- play / pause / resume / stop
- 현재 읽는 절 상태 관리
- 속도 관리
- unmount 시 speechSynthesis.cancel()

BibleChapterAudioControls
- 하단 컨트롤바의 듣기 버튼
- 재생 상태 표시
- 속도 선택
- 미지원 안내

VerseReader
- 현재 읽는 절 highlight 표시
```

성경 본문은 기존 `BibleVerse[]`를 그대로 사용한다.
새로운 식별자를 만들지 않고 기존 `verse_key` 정책을 유지한다.

```text
verse_key = 창1:1
```

하이라이트, 저장, 공유, 통독 완료와 TTS 상태는 서로 다른 관심사다.
TTS가 절을 읽었다고 해서 읽음 완료나 통독 완료로 처리하지 않는다.

## 데이터와 서버 정책

MVP에서는 백엔드 작업이 없다.

이유는 다음과 같다.

- 현재 성경 본문은 프론트 정적 JSON으로 제공된다.
- 브라우저 TTS는 서버 API가 필요 없다.
- 사용자별 저장 데이터가 아니다.
- 무료 서버 콜드스타트 문제를 피할 수 있다.

서버 TTS를 도입하는 경우에는 별도 Phase로 분리한다.
그때는 Spring Boot 백엔드에서 다음 항목을 검토한다.

- TTS provider 선택
- 음성 파일 생성 요청 API
- 파일 저장 위치
- 캐시 키: `translation_code + verse_key 또는 chapter_key + voice + rate`
- 비용 제한
- 저작권과 사용권 검토

## 라이선스 주의

WEB 본문은 Public Domain으로 확인했지만, 한국어 성경 본문은 별도 사용권 확인이 필요하다.
화면에 본문을 표시하는 권리와 본문을 음성으로 합성해 제공하는 권리가 동일하게 허용되는지 확인해야 한다.

MVP에서는 브라우저가 사용자의 기기에서 현재 화면의 텍스트를 읽는 방식으로 시작한다.
서버에서 음성 파일을 생성하거나 저장하는 방식은 사용권 검토 전까지 진행하지 않는다.

## 검증 계획

개발 후 최소 검증은 다음과 같다.

- Chrome 모바일 뷰에서 재생, 일시정지, 정지 확인
- Android Chrome 실제 기기 확인
- iOS Safari 또는 iPhone PWA 확인
- 장 이동 시 이전 음성 정지 확인
- 페이지 뒤로가기 시 음성 정지 확인
- 다크 모드와 라이트 모드에서 미니 플레이어 확인
- 작은 글, 큰 글에서 현재 절 표시 확인
- TTS 미지원 브라우저 안내 확인

`npm run build`는 사용자가 직접 실행하는 흐름을 유지한다.
AI는 변경 파일 단위 lint 또는 type check를 우선 확인한다.

## 후속 이슈 분리

문서 작성 후 구현은 다음 단위로 나누는 것을 추천한다.

```text
1. [feat/fe] 성경 읽기 TTS MVP 구현
2. [test/fe] TTS 모바일 브라우저 QA
3. [docs/ai] TTS 라이선스와 서버 확장 검토
```

첫 구현 이슈는 Web Speech API만 다룬다.
서버 TTS, 오디오 성경, 자동 연속 재생은 첫 구현 이슈에 포함하지 않는다.

## 결정 요약

- Phase 14는 TTS 듣기 기능을 다룬다.
- MVP는 브라우저 내장 TTS를 사용한다.
- 백엔드 작업은 없다.
- 현재 장 단위 듣기만 제공한다.
- 통독 완료와 TTS 재생 상태는 연결하지 않는다.
- 서버 TTS와 오디오 성경은 후속 Phase로 미룬다.
