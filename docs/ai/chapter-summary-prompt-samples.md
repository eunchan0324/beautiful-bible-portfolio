# 장별 요약 프롬프트 샘플 검수

> 관련 이슈: #83 `[docs/ai] 장별 요약 프롬프트 샘플 검수`
> 목적: 전체 장 요약 생성 배치 전에 프롬프트, 문체, 길이, 검수 기준을 먼저 확정한다.

---

## 목표

장별 요약은 성경 읽기 화면의 `AI 장 요약` 카드에 들어가는 짧은 보조 콘텐츠다. 사용자가 본문을 읽기 전에 그 장의 흐름을 빠르게 파악하도록 돕되, 본문 해석을 대신하지 않는다.

이번 문서의 목표는 다음 배치 작업으로 넘어가기 전에 아래 기준을 정하는 것이다.

- 어떤 입력을 LLM에 줄지
- 어떤 형식으로 결과를 받을지
- 어떤 문체와 길이를 사용할지
- 어떤 결과를 통과/수정/폐기할지

---

## 생성 원칙

- 사용자 요청 시 LLM을 호출하지 않는다.
- LLM은 운영자가 장 요약을 사전 생성할 때만 사용한다.
- LLM의 내장 성경 지식이 아니라, 앱에서 사용하는 운영 한국어 장 본문 전체를 입력으로 넣어 생성한다.
- 생성 결과는 먼저 `chapter_summary_candidates`에 저장하고, 승인된 결과만 `chapter_summaries` 테이블에 반영한다.
- 저장 후 앱에서는 DB 조회 결과만 표시한다.
- 장 요약은 해설, 묵상, 설교가 아니라 본문 흐름 안내다.

---

## 출력 기준

| 항목 | 기준 |
| --- | --- |
| 길이 | 50~80자 권장 |
| 문장 수 | 1문장 |
| 어조 | 담백한 설명체 |
| 종결 | `~입니다`, `~보여 줍니다`, `~다룹니다`, `~나타납니다` 권장 |
| 관점 | 운영 한국어 본문 흐름 기준 |
| 금지 | 적용 권면, 설교체, 교파 논쟁, 과도한 신학 해석 |

좋은 요약은 읽는 사람이 본문을 더 읽고 싶게 만드는 정도로 충분하다. 요약이 본문보다 앞서거나 결론을 단정적으로 밀어붙이면 수정한다.

## 프롬프트 초안

### System Prompt

```text
당신은 한국 개신교 독자를 위한 성경 본문 안내자입니다.
당신의 역할은 성경 본문을 해석하거나 설교하는 것이 아니라, 한 장의 내용 흐름을 짧고 정확하게 안내하는 것입니다.
신학적 논쟁이 있는 장은 교리적 결론을 미리 단정하지 말고, 본문에 나타난 서사와 묘사 자체를 우선합니다.
```

### User Prompt Template

```text
다음 성경 본문의 내용 흐름을 정확히 1문장으로 요약하여 JSON 형식으로 반환하세요.

입력:
- 책: {bookName} ({bookCode})
- 장: {chapterNum}장
- 본문(운영 한국어 본문): {chapterText}

핵심 규칙:
1. 문체: 담백하고 객관적인 설명체로 작성합니다.
2. 길이: 공백 포함 60자 내외를 목표로 하고, 최소 50자~최대 80자 범위에 맞춥니다.
3. 관점: 제공된 운영 한국어 본문에 직접 드러난 내용 흐름만 요약합니다.
4. 시작: 문장은 본문의 핵심 주어, 사건, 장면으로 시작합니다. 예: "하나님이", "예수님이", "이스라엘이"
5. 종결: 문장은 `~입니다`, `~보여 줍니다`, `~다룹니다`, `~나타납니다` 중 하나에 가까운 담백한 설명으로 끝냅니다.
6. 신학적 주의: 논쟁이 있는 본문은 구속사적 결론이나 교리 체계를 앞세우지 말고, 본문에 나타난 서사와 묘사 자체를 요약합니다.
7. 금지: 독자에 대한 권면, 묵상 적용, 설교적 표현을 사용하지 않습니다. 예: "~해야 합니다", "~합시다", "우리도"
8. 출력: 설명, 노트, 마크다운 코드 블록 없이 순수 JSON 객체만 반환합니다.

출력 형식:
{
  "summary": "요약 문장"
}
```

### 출력 예시

```json
{
  "summary": "하나님이 천지를 창조하시고 세상을 질서 있게 세우시는 장입니다."
}
```

## 샘플 장 선정

샘플은 장르와 난이도를 섞어 고른다. 창조, 타락, 율법, 시, 예언, 예수님의 가르침, 비유, 신학적 본문, 사랑의 권면을 모두 포함해 프롬프트가 한쪽 장르에만 맞춰지지 않도록 한다.

| 책 | 장 | 선정 이유 |
| --- | ---: | --- |
| 창세기 | 1 | 창조 서술, 현재 UI 샘플 데이터 |
| 창세기 | 3 | 타락 본문, 신학적 단정 주의 |
| 출애굽기 | 20 | 십계명, 율법 요약 톤 확인 |
| 시편 | 23 | 시 본문, 감정적 과장 주의 |
| 이사야 | 53 | 예언/고난 본문, 해석 단정 주의 |
| 마태복음 | 5 | 산상수훈, 긴 가르침 압축 |
| 누가복음 | 15 | 비유 묶음, 흐름 요약 |
| 요한복음 | 1 | 신학적 밀도 높음, 표현 정확성 확인 |

## 검수 기준

### 통과

- 본문에 있는 내용을 벗어나지 않는다.
- 1문장으로 자연스럽게 읽힌다.
- UI 카드에서 2~3줄 안에 들어간다.
- 신학적으로 민감한 본문을 과하게 단정하지 않는다.
- 사용자가 본문을 읽기 전에 흐름을 잡는 데 도움이 된다.

### 수정

- 80자를 크게 넘는다.
- 설교체나 적용 권면처럼 들린다.
- 특정 교파 해석처럼 보이는 표현이 있다.
- 너무 추상적이라 어떤 장인지 구분하기 어렵다.
- `하나님`, `예수님`, 인물 관계를 부정확하게 표현한다.

### 폐기

- 본문에 없는 내용을 만들어 낸다.
- 장 전체가 아니라 일부 구절만 요약한다.
- 신학적 논쟁 지점을 단정한다.
- 독자에게 행동을 지시한다.
- 감상적 표현이 본문 정보보다 앞선다.

## 나쁜 결과 예시

| 예시 | 문제 |
| --- | --- |
| `우리도 창조주 하나님을 믿고 매일 순종해야 합니다.` | 묵상 적용/권면에 가까움 |
| `창세기 3장은 원죄 교리를 완성하는 핵심 장입니다.` | 교리 해석을 과하게 단정 |
| `시편 23편은 힘든 현대인에게 위로를 주는 최고의 힐링 시입니다.` | 현대적 감상 표현이 과함 |
| `요한복음 1장은 삼위일체 교리를 자세히 설명합니다.` | 본문보다 신학 체계를 앞세움 |
| ```` ```json {"summary":"하나님이 천지를 창조하시는 장입니다."} ``` ```` | 마크다운 코드 블록이 섞여 파싱 실패 가능 |

## 자동 검증 규칙

배치 파이프라인은 사람 검수 전에 1차 자동 검증을 수행한다. 자동 검증은 신학적 정확성을 보장하지 않지만, 명백한 형식 오류와 문체 이탈을 줄인다.

| 검증 | 기준 | 실패 시 처리 |
| --- | --- | --- |
| JSON 파싱 | 순수 JSON 객체로 파싱 가능해야 함 | 재요청 또는 실패 기록 |
| 필드 존재 | `summary` 필수 | 재요청 또는 실패 기록 |
| 빈 응답 | `summary.trim().length > 0` | 재요청 |
| 길이 | 공백 포함 40자 미만 또는 120자 초과 시 실패 | 재요청 또는 검수 플래그 |
| 금지어 | `해야 합니다`, `합시다`, `우리도`, `성도`, `독자`, `묵상` 포함 시 실패 | 재요청 또는 검수 플래그 |
| 시작 표현 | `이 장은`, `본 장은`으로 시작하면 실패 | 재요청 또는 수정 |

종결어미 검증은 너무 강하게 적용하지 않는다. 장르별 자연스러운 문장을 허용하되, 설교체나 권면체로 흐르는 결과는 검수에서 수정한다.

## 배치 파이프라인으로 넘길 결정사항

- 입력은 `bookCode`, `bookName`, `chapterNum`, `chapterText`를 사용한다.
- `chapterText`는 앱에서 사용하는 운영 한국어 장 본문 전체를 그대로 전달한다.
- 출력은 `summary`만 포함한 JSON으로 받는다.
- 저장 전 JSON 파싱, 필드 존재, 빈 응답, 길이, 금지어, 시작 표현을 검증한다.
- `model` 컬럼에는 실제 API 모델명을 저장한다.
- 404/미생성 상태는 FE에서 카드 숨김으로 처리한다.
- 첫 배치에서는 샘플 장만 저장하고, 검수 통과 후 전체 1,189장으로 확장한다.
