# #125 하단 네비게이션 라우트 전환 체감 속도 개선

## 배경

하단 네비게이션에서 탭마다 이동 체감 속도가 달랐다.

- `/bible`은 로컬 데이터 중심이라 빠르게 전환됐다.
- `/ai`, `/my`는 첫 진입 때 약간 느리게 느껴졌다.
- `/search`는 배포 환경에서 두 번째 이후 진입에도 약 0.5초 정도 둔하게 느껴졌다.

처음에는 Render 무료 서버의 cold start를 의심했지만,
검색 화면은 첫 진입 시 검색 API를 바로 호출하지 않는다.
따라서 서버 API보다 프론트 라우팅과 페이지 설정을 먼저 확인했다.

## 원인

하단 네비게이션은 `button`에서 `router.push()`를 호출하는 구조였다.

```tsx
<button onClick={() => router.push(item.path)}>
```

이 방식은 페이지 이동은 가능하지만 Next.js의 `Link` 기반 자동 prefetch 이점을 얻기 어렵다.
사용자가 탭을 누르는 순간에야 해당 라우트 준비가 시작되므로 첫 이동이 느리게 느껴질 수 있다.

또한 `/search` 페이지에는 다음 설정이 있었다.

```tsx
export const dynamic = 'force-dynamic';
```

현재 검색 화면은 사용자가 검색어를 입력하기 전까지 API를 호출하지 않는 클라이언트 화면이다.
그런데 `force-dynamic`은 페이지를 매 요청마다 동적으로 처리하도록 강제하므로,
배포 환경에서 검색 탭만 계속 둔하게 느껴지는 원인 후보가 될 수 있었다.

## 결정

- 하단 네비게이션을 `button + router.push()`에서 `Link` 기반 이동으로 변경했다.
- `Link`에 `prefetch`를 명시해 하단 탭 라우트를 미리 준비할 수 있게 했다.
- `/search`의 `force-dynamic` 설정을 제거했다.
- 검색 API 호출 흐름은 바꾸지 않았다.

## 핵심 개념

`prefetch`는 사용자가 누르기 전에 다음 화면에 필요한 라우트 리소스를 미리 준비하는 방식이다.
API 자체를 빠르게 만드는 기능은 아니다.

```text
라우트 셸 로딩이 느림
→ Link prefetch로 개선 가능

페이지 내부 API가 느림
→ prefetch만으로 해결되지 않음
```

이번 작업은 백엔드 지연을 없애는 작업이 아니라,
탭을 눌렀을 때 화면 전환 자체가 늦게 시작되는 문제를 줄이는 작업이다.

## 변경 흐름

하단 네비게이션:

```text
button
→ onClick router.push
→ Link href
→ Next.js 자동 prefetch 활용
```

검색 페이지:

```text
force-dynamic 제거
→ 정적 라우트처럼 prefetch/cache 이점을 받을 수 있게 함
```

## 트레이드오프

`Link`로 전환해도 `/my` 내부의 사용자 정보 조회나 저장 말씀 개수 조회처럼 페이지 안에서 실행되는 API 지연은 그대로 남을 수 있다.
따라서 이후에도 느린 화면이 있다면 라우트 로딩과 데이터 로딩을 분리해 봐야 한다.

검색 화면의 `force-dynamic` 제거는 현재 구조에서는 적절하지만,
나중에 검색 페이지가 서버에서 요청별 데이터를 읽는 구조로 바뀐다면 다시 검토해야 한다.

## 왜 처음부터 Link로 만들지 않았는가

이 작업에서 중요한 질문은 "`Link`가 더 적절하다면 왜 처음부터 그렇게 만들지 않았을까?"였다.

답은 기술적 제약보다는 관성과 멘탈 모델에 가깝다.

- React에서는 "누르면 무언가 한다"는 상황에서 `onClick` 이벤트 핸들링이 먼저 떠오르기 쉽다.
- 하단 네비게이션은 시각적으로 아이콘과 라벨이 있는 버튼처럼 보이므로 `button`으로 생각하기 쉽다.
- `router.push()`도 동작 자체는 정상이라 처음에는 문제가 잘 드러나지 않는다.
- 페이지가 작고 사용자가 적을 때는 `Link` prefetch와의 체감 차이가 작아 보여 그냥 지나가기 쉽다.

하지만 하단 네비게이션처럼 단순 페이지 이동이 목적이라면 `Link`가 더 적절하다.
`Link`는 페이지 이동이라는 의미가 명확하고, Next.js의 자동 prefetch 흐름을 활용할 수 있다.

반대로 다음 경우에는 `button + router.push()`가 더 자연스러울 수 있다.

- 이동 전 로그인 여부나 권한을 검사해야 하는 경우
- 이동 전 상태 저장이 필요한 경우
- 분석 이벤트나 복잡한 클릭 로직을 먼저 처리해야 하는 경우

이번 하단 네비게이션은 그런 추가 로직이 없는 단순 이동이므로 `Link` 전환이 맞는 선택이었다.

이 결정은 "작동하는 코드"와 "상황에 더 맞는 코드"가 다를 수 있다는 점을 보여준다.
AI가 만든 코드도 그대로 받아들이지 않고, 프레임워크의 권장 흐름과 실제 사용자 체감을 기준으로 다시 검토해야 한다.

## 포트폴리오 포인트

- 사용자가 느끼는 "느림"을 백엔드 문제로만 보지 않고 라우팅, prefetch, 페이지 동적 설정으로 분해했다.
- `router.push()` 기반 명령형 이동을 `Link` 기반 선언형 이동으로 바꿔 접근성과 Next.js 최적화 흐름을 함께 개선했다.
- `force-dynamic`의 비용과 필요 조건을 검토해 불필요한 동적 렌더링 설정을 제거했다.
- 라우트 prefetch와 API 데이터 로딩의 차이를 구분해, 개선 범위와 한계를 명확히 정리했다.
- AI가 생성한 "동작은 하는 코드"를 그대로 두지 않고, 프레임워크 특성과 사용자 체감 성능 관점에서 리뷰하고 개선했다.
