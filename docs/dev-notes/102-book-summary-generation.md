# 102. 책 요약 생성 프롬프트와 66권 데이터 준비

## 배경

장 요약은 본문을 읽는 화면에서 한 장의 흐름을 빠르게 보여 주는 기능이다. 책 요약은 그보다 앞단에서, 사용자가 어떤 책을 읽으려는지 판단하고 큰 흐름을 잡는 안내 콘텐츠다.

이번 작업에서는 책 요약을 단순한 긴 문단 하나로 두지 않고, 화면 용도에 맞게 필드를 나누었다.

- `shortSummary`: 장 선택 화면 상단에서 가볍게 보여 줄 요약
- `summary`: 책 상세 화면에서 읽을 설명
- `readingPoint`: 읽을 때 붙잡을 관점
- `keywords`: 핵심 키워드
- `outline`: 책의 큰 흐름

## 결정

66권 전체 데이터는 운영자가 Supabase SQL Editor에서 실행할 수 있는 upsert SQL로 준비한다. `book_code`에 유니크 제약이 있으므로 같은 책 요약을 다시 실행해도 새 행이 중복 생성되지 않고 기존 내용이 갱신된다.

`keywords`와 `outline`은 MVP에서는 `TEXT` 컬럼에 JSON 문자열로 저장한다. 조회 API에서는 백엔드가 이를 `List<String>`으로 변환해 내려 주기 때문에 FE는 별도 파싱 없이 배열처럼 사용할 수 있다.

gpt-5-nano로 창세기 샘플을 생성해 보았고, JSON 구조 준수와 필드 구성은 안정적이었다. 다만 띄어쓰기 오류와 일부 추상적인 표현이 있어, 66권 전체는 AI 보조로 정리한 수동 검토 초안을 우선 사용하기로 했다. 따라서 `model` 값은 실제 자동 생성 모델명이 아니라 `ai-assisted-manual-review`로 기록한다.

이 결정은 AI를 쓰지 않았다는 뜻이 아니라, 생성 결과를 그대로 서비스 데이터로 넣지 않고 제품 문장 품질을 기준으로 비교해 선택했다는 의미다. 책 요약은 66개뿐이라 장 요약처럼 후보 테이블과 대량 배치 검수 구조를 만들기보다, 정적 upsert SQL과 샘플 QA가 더 단순하고 운영 비용이 낮다.

## 확인 방법

1. Supabase에서 `docs/sql/book-summaries-upsert.sql` 실행
2. `SELECT COUNT(*) FROM book_summaries;` 결과가 66인지 확인
3. Swagger에서 `GET /api/v1/bible/books/{bookCode}/summary` 확인
4. 샘플 책은 `창`, `레`, `시`, `마`, `계`를 우선 확인

## 포트폴리오 포인트

생성형 AI 콘텐츠를 그대로 화면에 붙이지 않고, 화면 용도에 맞게 데이터 구조를 먼저 나누었다. 또한 모델 생성 샘플과 수동 검토 초안을 비교한 뒤, 책 요약은 정적 upsert 방식이 더 적합하다고 판단했다. 이후 FE에서 같은 API를 장 선택 화면과 상세 화면에 재사용할 수 있게 설계했다.
