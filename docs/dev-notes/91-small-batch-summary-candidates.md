# 91. 장별 요약 후보 소규모 배치 검증

## 배경

장별 AI 요약은 바로 서비스 테이블에 저장하지 않고 `chapter_summary_candidates`에 먼저 저장한 뒤, AI 검수 결과와 운영 승인 상태를 분리해서 관리한다. 이번 작업은 이 구조가 실제 소규모 배치에서 동작하는지 확인하는 단계였다.

## 확인한 점

- 생성 모델은 `gemini-2.5-flash`, 검수 모델은 `llama-3.3-70b-versatile` 조합으로 테스트했다.
- Gemini 무료 티어는 분당 요청 수 제한이 있어, 여러 장을 연속 처리하면 429가 발생할 수 있다.
- 그래서 `AI_CHAPTER_SUMMARY_REQUEST_DELAY_MILLIS` 설정을 추가해 장 사이에 대기 시간을 둘 수 있게 했다.
- 재시도는 요청 수를 빠르게 소모하므로, 무료 티어 검증에서는 `AI_CHAPTER_SUMMARY_MAX_ATTEMPTS=1`이 더 안정적이다.

## 길이 기준 조정

기존 검증 기준은 `40~85자`였지만, 실제 창세기 테스트에서 자연스러운 한국어 요약이 85자를 넘는 경우가 자주 나왔다. 예를 들어 홍수, 계보, 언약처럼 사건이 많은 장은 85자 안에 담으려 하면 문장이 부자연스럽거나 정보가 과하게 빠졌다.

그래서 요약 길이 기준을 `40~120자`로 완화했다. 이 변경 후 이전에 길이 때문에 REVIEW가 되었던 창세기 7장, 9장 후보가 PASS로 바뀌었다.

## 운영 흐름

1. 배치가 후보를 생성한다.
2. AI 검수 결과가 `PASS`면 승인 후보로 본다.
3. `REVIEW`는 사람이 보류하거나 재생성한다.
4. 운영자가 SQL로 `PASS + PENDING` 후보만 `chapter_summaries`에 반영한다.
5. 반영된 후보는 `APPROVED` 상태로 변경한다.

이번 검증에서 창세기 13장 요약이 최종 테이블에 반영되었고, 기존 조회 API로 정상 응답되는 것을 Swagger에서 확인했다.

```json
{
  "bookCode": "창",
  "chapterNum": 13,
  "summary": "애굽에서 돌아온 아브람과 롯의 소유가 많아지자, 아브람은 롯에게 먼저 땅을 선택하게 하고 자신은 가나안 땅에 머무르며 하나님께 약속의 말씀을 받는 내용을 다룹니다.",
  "model": "gemini-2.5-flash"
}
```

## 정리

이번 단계의 핵심은 AI 생성 품질 자체보다, 무료 API 제한과 검수 품질 기준을 실제 데이터로 조정한 점이다. 후보 테이블을 둔 덕분에 실패나 애매한 문장을 서비스 데이터에 바로 넣지 않고, PASS 후보만 점진적으로 승인할 수 있었다.
