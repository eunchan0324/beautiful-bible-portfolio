# #120 백엔드 콜드스타트 대기 안내 및 중복 요청 방지

## 배경

프론트엔드는 Vercel, 백엔드는 Render 무료 티어로 운영하고 있다.
Render 무료 티어는 일정 시간 요청이 없으면 서버가 잠들 수 있고, 첫 요청 시 다시 켜지는 데 1~3분 정도 걸릴 수 있다.

개발자는 기다리면 된다는 사실을 알지만, 일반 사용자는 긴 스켈레톤 화면이나 느린 버튼 반응을 보면 앱이 멈췄다고 느낄 수 있다.
또한 기다리는 동안 저장, 검색, 하이라이트 같은 버튼을 반복해서 누르면 API 요청이 누적되고, 백엔드 rate limit에 걸려 `429 TOO_MANY_REQUESTS`가 발생할 수 있었다.

이번 작업은 서버 성능 자체를 바꾸는 대신, 현재 배포 환경의 제약을 사용자에게 명확히 설명하고 불필요한 중복 요청을 줄이는 것을 목표로 했다.

## 결정

- 홈 화면에서 백엔드 health check를 호출해 서버 준비 상태를 감지한다.
- 첫 접속 시 이미지 상단에 준비 상태 카드를 보여준다.
- 안내 문구는 "조금 걸릴 수 있어요"처럼 모호하게 쓰지 않고, 실제 체감 시간인 `1~3분`을 명시한다.
- 성경 읽기는 로컬 데이터 기반으로 바로 사용할 수 있으므로, 준비 카드에서도 "성경 읽기는 바로 볼 수 있다"는 점을 먼저 알려준다.
- 검색, 탐색, 마이페이지처럼 API 조회가 필요한 화면은 로딩이 길어질 때 공통 안내 UI를 보여준다.
- API 에러를 공통 `ApiError` 타입으로 정리해 `429`, `409` 같은 상태를 프론트에서 일관되게 처리한다.
- 저장/하이라이트 액션 중에는 선택 액션 버튼을 잠가 빠른 반복 클릭으로 같은 요청이 여러 번 나가지 않게 한다.
- 이미 저장한 말씀을 다시 저장할 때 발생하는 `409 CONFLICT`는 실패 문구 대신 "이미 저장한 말씀이에요"로 번역한다.

## 주요 흐름

홈 화면:

```text
홈 진입
→ /health 호출
→ 준비 중이면 안내 카드 표시
→ 10초 이상 걸리면 "아직 준비 중" 상태로 전환
→ health check 성공 시 "준비가 끝났어요" 표시 후 자동 숨김
```

API 에러 처리:

```text
fetch 응답 확인
→ response.ok가 아니면 ApiError 생성
→ status 429는 요청 과다 문구로 표시
→ status 409는 필요한 화면에서 중복 저장 문구로 표시
→ 그 외는 기존 fallback 문구 사용
```

저장 버튼 처리:

```text
말씀 선택
→ 저장 버튼 클릭
→ pending 상태로 버튼 잠금
→ 저장 API 호출
→ 전부 409면 "이미 저장한 말씀이에요"
→ 일부만 성공하면 성공 개수와 이미 저장된 항목 안내
→ 완료 후 pending 해제
```

## 트레이드오프

백엔드 콜드스타트 자체를 없애려면 유료 인스턴스, 외부 uptime monitor, 별도 warming job 같은 선택지가 있다.
하지만 현재 단계에서는 비용을 늘리는 대신, 사용자가 기다릴 이유를 이해할 수 있게 만들고 중복 요청을 줄이는 방식이 더 적절했다.

이 방식은 첫 요청 시간을 실제로 줄이지는 못한다.
대신 "고장난 것처럼 보이는 시간"을 "준비 중이라는 것을 이해할 수 있는 시간"으로 바꾸고, 사용자가 같은 작업을 여러 번 누르며 문제를 키우는 상황을 줄인다.

## 검증

- `npx.cmd tsc --noEmit --incremental false`
- 홈 준비 카드 문구 확인
- 느린 요청 안내 UI 임시 토글로 검색/탐색/마이페이지 화면 확인
- 저장 버튼 반복 클릭 시 요청이 연속 발생하지 않는지 확인
- 이미 저장한 말씀 재저장 시 `Data conflict occurred` 대신 사용자 문구가 보이는지 확인

## 포트폴리오 포인트

- 무료 배포 환경의 콜드스타트 제약을 제품 UX와 요청 제어로 완화했다.
- 서버 성능 문제를 단순히 "느림"으로 두지 않고, 사용자 안내, 공통 에러 타입, 중복 요청 방지로 나누어 단계적으로 해결했다.
- 성경 읽기처럼 즉시 가능한 기능과 API 준비가 필요한 기능을 구분해, 전체 앱을 막지 않는 degraded UX를 설계했다.
- `429 TOO_MANY_REQUESTS`와 `409 CONFLICT`를 사용자 행동에 맞는 문구로 번역해 API 상태 코드와 프론트 경험을 연결했다.
- 비용, 운영 환경, 사용자 신뢰 사이의 트레이드오프를 고려한 안정화 작업으로 설명할 수 있다.
