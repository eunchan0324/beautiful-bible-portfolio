# #121 하이라이트 반응 속도 및 API 동기화 안정화

## 배경

하이라이트 기능은 처음에는 localStorage에 바로 저장했다.
이 방식은 네트워크를 기다리지 않기 때문에 사용자가 색상을 누르는 즉시 화면에 반영되었다.

로그인 사용자 데이터를 백엔드 API로 옮기면서 하이라이트 저장/수정/삭제가 서버 요청을 거치게 되었다.
그 결과 API 응답이 끝난 뒤에야 화면 색상이 바뀌어, 특히 Render 무료 티어의 콜드스타트나 네트워크 지연 상황에서 기능이 느리거나 멈춘 것처럼 느껴질 수 있었다.

또한 로그인 상태에서 localStorage 하이라이트가 남아 있으면 성경 화면 진입 시 `/highlights/sync`가 반복 호출될 가능성이 있었다.
이 경우 사용자가 직접 누른 요청이 아니더라도 API 호출량이 늘어나고, rate limit에 가까워질 수 있었다.

## 결정

- 로그인 후 localStorage 하이라이트를 백엔드에 sync한 뒤에는 localStorage 데이터를 비운다.
- 하이라이트 색상 선택과 삭제는 화면 상태를 먼저 바꾸는 optimistic update 방식으로 처리한다.
- 서버 API 호출은 화면 반영 이후 백그라운드에서 실행한다.
- 여러 구절을 선택한 경우 `Promise.allSettled`로 요청을 병렬 처리한다.
- 일부 요청이 실패하면 전체 상태를 되돌리지 않고, 실패한 구절 중 아직 optimistic 상태가 유지되는 항목만 이전 상태로 롤백한다.
- 이번 단계에서는 백엔드 bulk API를 추가하지 않는다. 먼저 프론트엔드 호출 흐름만 개선한다.

## 주요 흐름

초기 하이라이트 로딩:

```text
로그인 상태 확인
→ localStorage 하이라이트 확인
→ 남아 있으면 /highlights/sync 호출
→ sync 성공 후 localStorage 비우기
→ 서버 기준 하이라이트 상태를 화면에 반영
```

하이라이트 색상 선택:

```text
구절 선택
→ 색상 선택
→ 화면 하이라이트 즉시 변경
→ 선택 액션바 닫기
→ create/update/delete API를 백그라운드 병렬 호출
→ 실패한 요청이 있으면 해당 구절만 이전 색상으로 롤백
```

하이라이트 삭제:

```text
구절 선택
→ 하이라이트 삭제
→ 화면에서 즉시 제거
→ delete API를 백그라운드 병렬 호출
→ 실패한 요청이 있으면 해당 구절만 이전 색상으로 롤백
```

## 트레이드오프

백엔드에 bulk upsert/delete API를 추가하면 여러 구절 하이라이트를 더 깔끔하게 한 번의 요청으로 처리할 수 있다.
하지만 이번 이슈의 핵심은 API 스펙 확장이 아니라, API 전환으로 느려진 사용자 반응을 먼저 회복하는 것이었다.

따라서 이번 단계에서는 기존 API를 유지하고 프론트엔드에서 optimistic update와 병렬 요청으로 체감 속도를 개선했다.
이 방식은 서버 요청 수 자체를 한 번으로 줄이지는 못하지만, 사용자가 기다리는 시간을 줄이고 실패 상황도 화면 상태가 깨지지 않게 처리할 수 있다.

## 검증

- `npx.cmd tsc --noEmit --incremental false`
- 로그인 상태에서 하이라이트 색상 선택 시 즉시 화면에 반영되는지 확인
- 여러 구절 선택 후 색상 선택 시 액션바가 바로 닫히는지 확인
- 기존 `npm run lint`는 `next.config.ts`의 `require()` 에러와 generated service worker 경고로 실패한다. 이번 변경 파일에서 새 lint 에러는 확인되지 않았다.

## 포트폴리오 포인트

- localStorage 기반 빠른 UX를 API 저장소 전환 이후에도 유지하기 위해 optimistic UI를 적용했다.
- 네트워크 지연과 서버 콜드스타트가 사용자 액션을 막지 않도록 화면 반응과 서버 저장을 분리했다.
- 실패 시 전체 상태를 되돌리지 않고 실패한 항목만 롤백해 부분 실패에 강한 UI 상태 관리를 설계했다.
- localStorage sync 후 데이터를 비워 반복 동기화와 불필요한 API 호출을 줄였다.
- 백엔드 bulk API를 바로 추가하지 않고, 기존 API 계약 안에서 먼저 체감 성능을 개선하는 단계적 접근을 선택했다.
