## 목적

Phase 4의 키워드 검색(PostgreSQL FTS)을 보완해, **자연어 의미 기반 검색**을 도입한다.

- 키워드 검색: "사랑"이라는 **단어가 들어간** 구절 찾기
- 의미 검색: "위로가 필요할 때" → **의미적으로 관련된** 구절 찾기 (단어 일치 X)

레퍼런스: 초원 앱의 "맥락 찾기" (예: "성령의 9가지 열매")

## 방향성 (결정됨)

### Retrieval-first, NOT a chatbot

- 챗봇 형태 X (ChatGPT/Perplexity 차별화 어려움)
- "내 상황에 맞는 말씀을 찾아주는 검색기" 포지셔닝
- LLM 해설 기능은 Phase 7b 선택 사항으로 미룸

### 왜 이 방향인가

- 사용자(교회 성도) 입장: 자유 대화보다 "관련 구절 추천"이 실질적 가치
- 포트폴리오 입장: "RAG의 R(Retrieval)을 제대로 만들었다"가 "LLM 한 번 호출했다"보다 임팩트
- 운영 입장: LLM 호출 없음 = 비용 거의 0

## 아키텍처 결정 (결정됨)

### 외부 임베딩 API + 인터페이스 추상화

```java
interface EmbeddingService {
    float[] embed(String text);
}
```

- 구현체는 갈아끼울 수 있게 (OpenAI / Voyage / Upstage / 향후 자가 호스팅)
- 모델명, 벡터 차원은 `application.yml`로 분리

### 왜 로컬 LLM이 아닌가

- Render 무료 티어 512MB RAM — BGE-M3 등 임베딩 모델 자가 호스팅 불가
- 자가 호스팅 = 사용자 100명 단계에 오버엔지니어링
- 트래픽 1만+ 단계에서 비용 역전 시 마이그레이션 (인터페이스 추상화로 대비)

### 호출 흐름

```
[사전 작업, 한 번]
31,089 구절 → 외부 임베딩 API → pgvector DB 저장

[검색 시점, 매 요청]
사용자 쿼리 → 외부 임베딩 API (쿼리 벡터 1개 생성)
            → pgvector 코사인 유사도 검색 (수학, AI 아님)
            → 관련 구절 N개 반환
```

### 비용 예상

OpenAI text-embedding-3-small 기준:
- 31,089 구절 사전 임베딩: 약 30원 (1회)
- 사용자 1명 월 100회 검색: 약 0.03원
- 교회 50명이 월 100회 검색: **약 1.5원/월**

## 결정 필요 항목

### 1. 임베딩 모델 선정 (실험 필요)

후보:

| 모델 | 한국어 성능 | 무료 티어 | 비고 |
| ---- | ---------- | -------- | ---- |
| OpenAI text-embedding-3-small | ★★★★ | 없음 (저렴) | 업계 표준, SDK 풍부 |
| Voyage voyage-3-lite | ★★★★ | 50M 토큰 | Anthropic 공식 파트너 |
| Upstage Solar Embedding | ★★★★★ | 무료 토큰 | 한국 회사, 한국어 1위 |
| Cohere embed-multilingual-v3 | ★★★★ | trial key | 다국어 강점 |

실험 방법:
- 동일 쿼리 세트로 두세 모델 결과 비교
- 쿼리 예시: "위로", "감사", "용기", "두려움", "사랑", "성령의 열매"
- 정성 평가 (성도에게 "이 결과가 적절한가" 묻기)

### 2. Chunking 전략

옵션:
- A. **구절 단위 (1절씩)** — 31,089개. 가장 단순, 검색 결과 단위 명확
- B. **단락 단위 (여러 절 묶음)** — 의미 단위로 묶음. 더 풍부한 컨텍스트
- C. **장 단위** — 검색 결과가 너무 큼. 부적합

학습 포인트: chunk 크기와 검색 품질의 트레이드오프

### 3. 검색 결과 평가 방법

- **정성 평가:** 쿼리 세트별 top-5 결과를 직접 보고 평가
- **정량 평가:** "정답 구절 집합" 미리 정의 → Recall@5 측정
- **A/B 비교:** 키워드 검색 vs 의미 검색 결과 나란히 보고 차이 확인

### 4. 응답 DTO 설계

- 유사도 점수 노출 여부 (디버깅용 vs 사용자 노출)
- top-N 개수 (5? 10?)
- 페이징 필요 여부

### 5. FastAPI 보조 서버 도입 여부

- 현 단계: Spring Boot가 직접 외부 API 호출 (단순)
- 향후: 자가 호스팅 임베딩 모델 도입 시 FastAPI 보조 서버 검토

### 6. Phase 7b: LLM 해설 도입 시점

- "이 구절들이 왜 당신 상황에 맞는지" 짧은 해설
- Claude Haiku / Gemini Flash 등 저비용 LLM
- Phase 7a(검색)가 안정화된 후 검토

## 미래 확장 고려사항

### 트래픽 단계별 인프라 전략

| 사용자 규모 | 임베딩 방식 | 인프라 |
| ---------- | ---------- | ------ |
| ~100명 (현재) | 외부 API | Render 무료 |
| ~1,000명 | 외부 API 유지 | Render 유료 검토 |
| ~10,000명+ | 자가 호스팅 검토 | 별도 GPU/CPU 서버 |

→ 후원금 우선순위: DB > 백엔드 서버 > 임베딩 인프라

### 다른 AI 기능 확장 시 동일 패턴

- TTS (성경 낭독): TtsService 인터페이스 + 외부 API
- 묵상 추천: RecommendationService 인터페이스
- 모든 AI 기능을 인터페이스로 격리 → 구현체 교체 자유

## 면접 포인트

- 왜 챗봇이 아닌 retrieval-first인지: 차별화 + 비용 + 사용자 가치
- 왜 외부 API 시작인지: 트래픽 단계별 최적 방식 / 추상화로 마이그레이션 가능
- 임베딩 모델 비교 실험 과정: 한국어 도메인 평가 방법
- Chunking 전략 결정 과정: 검색 품질 vs 결과 단위 트레이드오프
- pgvector 선택 이유: 별도 벡터 DB(Pinecone/Qdrant) 대비 운영 단순화
- EmbeddingService 인터페이스: SOLID의 DIP 적용 사례
