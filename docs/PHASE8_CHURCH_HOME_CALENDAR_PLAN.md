# Phase 8: 우리 교회 홈 & 일정 캘린더 MVP

> Phase 7 이후 방향 전환 문서.
> 기존 읽기 출석/누적 통계 중심 계획은 교회 타겟과 통독 확장 방향에 맞춰 제거하고,
> 우리 교인이 실제로 다시 열 이유가 있는 홈과 일정 캘린더를 먼저 만든다.

---

## 배경

현재 앱은 성경 읽기, 검색, 해설, 저장/하이라이트 기능을 갖췄지만,
타겟 사용자를 "우리 교회 교인"으로 좁히면 매주 다시 열 이유가 아직 약하다.

회의에서 나온 후보는 다음과 같다.

- PWA 오늘의 말씀 알림
- 개인/단체 통독
- 교회 일정 알림
- 교회 전용 홈

이 중 교회 일정은 운영 공수가 생길 수 있으므로 처음부터 관리자 시스템을 크게 만들지 않는다.
대신 홈과 캘린더를 작게 만들고, 일정 데이터는 당분간 개발자가 직접 관리할 수 있는 정적 데이터로 시작한다.

---

## 결정

Phase 8은 다음 범위로 진행한다.

```text
우리 교회 홈 & 일정 캘린더 MVP
```

포함한다.

- 홈 화면을 우리 교회 교인이 쓰는 대시보드 방향으로 정리
- 오늘의 말씀 유지
- 다가오는 교회 일정 표시
- 교회 일정 캘린더 화면 추가
- 일정 데이터는 정적 TS/JSON 데이터로 시작

포함하지 않는다.

- 출석 체크 캘린더
- 누적 읽기 통계
- 관리자 일정 등록/수정 화면
- PWA 일정 알림
- Google Calendar/ICS 연동
- 개인/단체 통독 진행률

---

## 왜 출석 캘린더를 제외하는가

출석 체크는 다른 앱에도 있고 구현하기 쉬운 기능이지만,
현재 제품 목표에서는 단독 가치가 약하다.

- "앱을 켰다" 또는 "성경 1장을 읽었다"를 출석으로 정의하는 기준이 애매하다.
- 개인/단체 통독이 들어오면 읽음 완료와 진행률 기능이 자연스럽게 생긴다.
- 출석 캘린더를 따로 만들면 교회 일정 캘린더와 화면 개념이 겹친다.

따라서 읽기 기록은 지금 만들지 않고,
나중에 개인 통독 또는 단체 통독의 진행 데이터로 자연스럽게 다룬다.

---

## UX 방향

홈은 "성경 앱의 시작 화면"이면서 "우리 교회 생활을 확인하는 화면"이 된다.

예상 구성:

```text
홈
- 오늘의 말씀
- 성경 읽기 진입
- 다가오는 교회 일정
- 교회 캘린더 진입
- 이후: 통독 진행 상황
```

캘린더 화면:

```text
교회 캘린더
- 월 이동
- 일정이 있는 날짜 표시
- 날짜 선택 시 하단에 해당 날짜 일정 표시
```

---

## 데이터 전략

처음에는 백엔드 API를 만들지 않는다.

이유:

- Render 무료 서버의 cold start 영향을 받지 않는다.
- 일정 관리 담당자가 아직 없으므로 관리자 화면부터 만들 필요가 없다.
- 직접 관리할 일정 수가 적으면 정적 데이터가 가장 단순하다.
- 나중에 DB나 외부 캘린더로 옮기기 쉽도록 필드 구조만 미리 맞춘다.

예상 데이터:

```ts
export const CHURCH_EVENTS = [
  {
    id: '2026-06-07-sunday-service',
    title: '주일예배',
    date: '2026-06-07',
    time: '오전 11:00',
    location: '본당',
    type: 'worship',
  },
];
```

## 구현 순서

1. 홈 화면 현재 구조 확인
2. 교회 일정 데이터 파일 추가
3. 홈에 다가오는 일정 카드 추가
4. 교회 캘린더 라우트 추가
5. 월간 캘린더 UI 구현
6. 날짜별 일정 표시
7. 모바일 화면 QA
8. dev-note 기록

