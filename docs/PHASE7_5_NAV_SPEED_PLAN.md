# Phase 7.5: 하단 네비게이션 속도 개선

> 관련 이슈: #125
> 새 기능을 붙이기 전에 탭 이동 체감 속도를 먼저 안정화한다.

---

## 배경

하단 네비게이션에서 탭별 이동 속도가 다르게 느껴진다.

- `/bible`: 로컬 데이터 중심이라 빠름
- `/search`: 가장 느리게 느껴짐
- `/ai`, `/my`: 중간 정도로 느림

Render 무료 서버의 cold start 문제도 있지만, 탭 이동 자체는 프론트 라우팅, 페이지 번들, 초기 API 호출 흐름도 함께 확인해야 한다.

---

## 목표

- 탭별 속도 차이의 원인을 설명할 수 있게 정리한다.
- 하단 네비게이션 이동 시 체감 속도를 개선한다.
- 기능 동작은 기존과 동일하게 유지한다.
- 이후 Phase에서 홈/캘린더/통독 기능이 붙어도 기본 이동 경험이 흔들리지 않게 한다.

---

## 확인할 지점

### 하단 네비게이션

- `BottomNavigation`이 `router.push()` 기반인지 확인
- `Link` 기반 전환 또는 `router.prefetch()` 적용 가능성 검토
- 현재 활성 탭 계산이 불필요하게 무겁지 않은지 확인

### 검색 탭

- `/search`의 `dynamic = 'force-dynamic'` 필요 여부 확인
- 검색 페이지 첫 진입 시 실제 API 호출이 있는지 확인
- 검색 컴포넌트 번들/초기 렌더링 비용 확인

### 마이 탭

- 인증 상태 확인
- `users/me`, 저장 말씀 개수 조회 등 초기 API 호출 흐름 확인
- 첫 화면 전환과 데이터 로딩 상태를 분리할 수 있는지 확인

### 해설 탭

- 초기 화면에서 API 호출이 없는지 확인
- 이미지/컴포넌트 로딩 비용 확인

---

## 구현 후보

1. 하단 네비게이션 항목 prefetch
2. 가능한 탭은 `Link` 기반 이동으로 변경
3. `/search`의 `force-dynamic` 제거 가능성 검토
4. `/my` 첫 진입 시 API 결과를 기다리지 않고 기본 UI 먼저 표시
5. 느린 데이터는 각 화면 내부 skeleton/notice로 처리

---

## 완료 기준

- 탭별 속도 차이 원인을 문서 또는 dev-note에 정리한다.
- 최소 1개 이상의 체감 속도 개선을 적용한다.
- 적용하지 않은 후보가 있다면 이유를 남긴다.
- `npm run build` 또는 타입 체크로 기능 이상 여부를 확인한다.
